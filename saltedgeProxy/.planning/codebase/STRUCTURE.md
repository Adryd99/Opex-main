# Structure

## Top-Level Layout
- `pom.xml` defines a single-module Maven build for a Spring Boot 4 application on Java 17.
- `src/main/java` contains all production Java code under the base package `com.saltedgeproxy.app.saltedgeproxy`.
- `src/main/resources` contains `application.properties` plus currently empty `static/` and `templates/` directories.
- `src/test/java` contains a minimal Spring context smoke test in `src/test/java/com/saltedgeproxy/app/saltedgeproxy/SaltedgeproxyApplicationTests.java`.
- `data/` stores the file-backed H2 database files referenced by the datasource URL.
- `target/` contains compiled artifacts and should be treated as build output, not source.
- `.planning/codebase/` is reserved for repository reference docs like this one and `ARCHITECTURE.md`.

## Java Package Map
- `src/main/java/com/saltedgeproxy/app/saltedgeproxy/SaltedgeproxyApplication.java`: application bootstrap class.
- `src/main/java/com/saltedgeproxy/app/saltedgeproxy/controller/`: inbound REST adapters. Today this contains only `UserController.java`.
- `src/main/java/com/saltedgeproxy/app/saltedgeproxy/service/`: outbound integration logic. Today this contains only `SaltEdgeService.java`.
- `src/main/java/com/saltedgeproxy/app/saltedgeproxy/repository/`: Spring Data JPA repository interfaces for persisted entities.
- `src/main/java/com/saltedgeproxy/app/saltedgeproxy/model/`: JPA entities and their table mappings.
- `src/main/java/com/saltedgeproxy/app/saltedgeproxy/dto/`: request/response classes for Salt Edge JSON payloads.
- `src/main/java/com/saltedgeproxy/app/saltedgeproxy/config/`: startup-time Spring configuration, currently only database seeding.

## Key Modules
- `controller/UserController.java` is the main functional hub and currently combines endpoint definition, workflow orchestration, and persistence coordination.
- `service/SaltEdgeService.java` is the only external-system adapter and should remain the place for Salt Edge HTTP concerns.
- `model/User.java` is the central entity because it anchors the one-to-many relationships to `BankAccount`, `Transaction`, and `Tax`.
- `model/BankAccount.java` and `model/Transaction.java` model imported banking data and define the path used by the sync flow.
- `repository/UserRepository.java`, `repository/BankAccountRepository.java`, and `repository/TransactionRepository.java` provide the persistence API used by the controller.
- `config/DataInitializer.java` seeds representative users, accounts, and transactions for local development and demo flows.

## Important Resources
- `src/main/resources/application.properties` is the primary runtime configuration file.
- `README.md` describes the intended API surface and local run instructions.
- `HELP.md` is the default Spring Boot help file generated by the project template.
- `mvnw` and `mvnw.cmd` are the preferred Maven wrapper entry points for local builds.
- `data/testdb.mv.db` and related files are the current local H2 database artifacts produced by the configured datasource.

## Naming And Layout Conventions
- Package naming is fully lowercase and follows the Java package root `com.saltedgeproxy.app.saltedgeproxy`.
- File names are conventional Spring names: controller classes end in `Controller`, service classes end in `Service`, repository interfaces end in `Repository`, and DTOs are prefixed with `SaltEdge`.
- JPA entities live in `model/` and use table-oriented annotations like `@Table(name = "users")` or `@Table(name = "bank_account")`.
- Repository methods follow Spring Data derived-query naming such as `findByUser_Id(...)` and `findByConnectionId(...)`.
- External API field mismatches are normalized in DTOs with `@JsonProperty`, for example `customer_id`, `connect_url`, `currency_code`, and `made_on`.
- Lombok is the default boilerplate strategy across entities and DTOs, so new classes in these areas should follow the existing annotation style unless the project intentionally moves away from Lombok.

## Practical Placement Guidance
- New REST endpoints should go in `src/main/java/com/saltedgeproxy/app/saltedgeproxy/controller/` unless the codebase is first refactored toward feature-based packaging.
- New Salt Edge API operations should go in `src/main/java/com/saltedgeproxy/app/saltedgeproxy/service/SaltEdgeService.java` or a sibling service in the same package if the integration surface grows.
- New persistence queries should be added to the relevant interface in `src/main/java/com/saltedgeproxy/app/saltedgeproxy/repository/`.
- New database-backed domain types should be added under `src/main/java/com/saltedgeproxy/app/saltedgeproxy/model/`.
- New external transport objects should be added under `src/main/java/com/saltedgeproxy/app/saltedgeproxy/dto/` rather than mixed into entities.
- New startup or bean wiring logic should go in `src/main/java/com/saltedgeproxy/app/saltedgeproxy/config/`.
- Frontend assets or server-rendered templates would belong in `src/main/resources/static/` or `src/main/resources/templates/`, but those folders are currently unused.

## Current Structural Characteristics
- The repository is intentionally small and monolithic: one application module, one controller, one service, three active repositories, and a small entity set.
- Architectural boundaries exist by package, but behavior is not deeply decomposed; orchestration logic is concentrated in `UserController`.
- The project uses Spring Boot auto-configuration instead of explicit infrastructure modules.
- There is no separate `client`, `mapper`, `exception`, `security`, or `configuration-properties` package yet.
- Tests are skeletal, so structure currently communicates intent more through production code and `README.md` than through a rich test suite.
- `Tax.java` exists in the model layer without matching repository/controller usage, which signals planned or partial functionality.

## Where New Code Should Go First
- If the change is an API contract change, start in `controller/` and then push shared logic down instead of expanding controller methods indefinitely.
- If the change is about Salt Edge request composition or response handling, keep the HTTP details in `service/` and the JSON shape in `dto/`.
- If the change is about local storage or retrieval, update `model/` and `repository/` together so table mappings and access patterns stay aligned.
- If the change is purely configuration driven, update `application.properties` and, when typed access becomes necessary, add a dedicated config class under `config/`.
- If the feature introduces another bounded area beyond users and syncing, consider adding a new package peer to `controller`, `service`, `repository`, and `model` only after there is enough behavior to justify it.
